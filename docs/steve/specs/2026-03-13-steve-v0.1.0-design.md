# Steve v0.1.0 — Design Specification

## Overview

Steve is a stateful, opinionated development workflow tool for Claude Code. Named after Steve Rogers, it provides persistent project state and phase management that survives `/clear` and session boundaries — the critical gap in existing Claude Code tooling.

**Core value proposition:** Other tools handle *how* to work within a session. Steve handles *what* to work on across sessions, with automated review gates that prevent design-implementation drift.

**Non-goals:** Steve is not a plugin, not a framework, and does not depend on any third-party plugins (superpowers, etc.). It uses only Claude Code's built-in capabilities.

---

## Architecture

### Form Factor

- **`steve` CLI** — a bash script installed at `~/.local/bin/steve` with two subcommands: `steve init` (copies commands + launches Claude Code) and `steve update` (checks for new version, updates CLI + project commands)
- **8 slash commands** — markdown-based commands in `.claude/commands/` that drive the entire workflow inside Claude Code
- **State files** — persistent markdown + JSON in `docs/steve/` that survive session boundaries

### Command Naming

- CLI: `steve init`
- Slash commands: `/steve:init`, `/steve:design`, `/steve:prd`, `/steve:plan-phase`, `/steve:next-task`, `/steve:test-checkpoint`, `/steve:complete-phase`, `/steve:resume`

Colon-separated naming prevents collisions with Claude Code's built-in commands (e.g., `/resume` vs `/steve:resume`).

**Cross-platform note:** Colons in filenames are legal on Linux and macOS (HFS+/APFS treat `:` as `/` internally but allow it via terminal). On Windows/NTFS, colons are reserved. Since Claude Code primarily targets Linux and macOS, this convention is safe. If Windows support is needed in a future version, the files can be renamed to use a directory structure (`.claude/commands/steve/design.md` → `/steve/design`) without changing the workflow.

**Manual invocation:** `/steve:init` can be invoked directly inside Claude Code without the CLI. The `steve init` CLI is a convenience wrapper that handles git setup before launching Claude Code. Both entry points are supported.

## Workflow Pipeline

```
steve init → /steve:design → /steve:prd → [phase loop] → done
                                              │
                                    ┌─────────┴──────────┐
                                    │                     │
                              /steve:plan-phase           │
                                    │                     │
                              /steve:next-task ──┐        │
                                    ▲            │        │
                                    └────────────┘        │
                                    (repeat per task)     │
                                    │                     │
                              /steve:test-checkpoint      │
                                    │                     │
                              /steve:complete-phase ──────┘
                                    (repeat per phase)

  /steve:resume ← can re-enter at any point in the phase loop
```

### Command Details

| Command | Inputs | Outputs | Review Gate |
|---|---|---|---|
| `steve init` (CLI) | None | Launches Claude Code with `/steve:init` | None |
| `/steve:init` | Interactive Q&A | `config.json`, `CLAUDE.md`, `.gitignore`, project scaffold | None |
| `/steve:design` | Conversation + codebase | `docs/steve/design.md` | Sonnet: completeness, contradictions, gaps |
| `/steve:prd` | `design.md` | `docs/steve/prd.md` | Sonnet: all design features covered, no drift |
| `/steve:plan-phase` | `prd.md` + `config.json` (infers phase number) | `docs/steve/current-phase.md` | Sonnet: tasks cover full phase scope |
| `/steve:next-task` | `current-phase.md`, `config.json` | Code + tests, commits, task checkbox updated | None |
| `/steve:test-checkpoint` | `current-phase.md`, codebase | Pass/fail report | Sonnet: feature completeness against phase spec + PRD |
| `/steve:complete-phase` | Checkpoint passed | Archives phase, merges branch, sets up next | None |
| `/steve:resume` | All state files + git history | Context reconstruction + next action recommendation | None |

### Enforcement Rules (State Machine)

Each command checks `workflowState` before executing and refuses to run out of order:

- `init` sets state to `initialized`
- `design` requires state = `initialized`, sets state to `designed`
- `prd` requires state = `designed`, sets state to `specced`
- `plan-phase` requires state = `specced` or `phase-completed`, sets state to `phase-planned`
- `next-task` requires state = `phase-planned` or `in-progress`, sets state to `in-progress`
- `test-checkpoint` requires state = `in-progress` AND all task checkboxes marked `[x]` in `current-phase.md`, sets state to `checkpoint-passed`
- `complete-phase` requires state = `checkpoint-passed`, sets state to `phase-completed` (or `completed` if final phase)
- `resume` can run from any state, does not change state

**How `test-checkpoint` verifies all tasks are complete:** The command reads `current-phase.md` and parses lines matching `- [ ] ` or `- [x] ` **only within the `## Tasks` section** (from the `## Tasks` header until the next `##` header). The `## Test Checkpoint` section's checkboxes are NOT included in this count — those are managed by the checkpoint verification itself. If any unchecked tasks remain (`[ ]`) in the Tasks section, the command refuses to proceed and lists the incomplete tasks. This is a hard gate — no override.

## State Management

All state lives in `docs/steve/` as two primary files:

### `docs/steve/config.json`

```json
{
  "projectName": "my-app",
  "description": "...",
  "architecture": "fullstack",
  "backend": { "language": "python", "framework": "fastapi" },
  "frontend": { "language": "typescript", "framework": "nextjs" },
  "database": { "engine": "postgresql" },
  "testing": { "methodology": "TDD" },
  "git": {
    "branchPrefix": "phase",
    "commitStyle": "conventional"
  },
  "workflowState": "initialized",
  "currentPhase": null,
  "totalPhases": null,
  "completedPhases": [],
  "steveVersion": "0.1.0"
}
```

- `currentPhase`: integer, set by `plan-phase`
- `totalPhases`: integer, set by `prd` (total number of phases in the plan)
- `completedPhases`: array of integers, appended by `complete-phase`

### Workflow State Transitions

```
initialized → designed → specced → phase-planned → in-progress → checkpoint-passed → phase-completed
                                        ↑                                                  │
                                        └──────────────────────────────────────────────────┘
                                              (next phase, via /steve:plan-phase)

                                   phase-completed → completed (when currentPhase == totalPhases)
```

- `initialized` → `designed`: set by `/steve:design` after design doc + review pass
- `designed` → `specced`: set by `/steve:prd` after PRD + review pass. Also sets `currentPhase` to 1 and `totalPhases` to the number of phases defined in the PRD
- `specced` → `phase-planned`: set by `/steve:plan-phase` when expanding a phase into tasks
- `phase-planned` → `in-progress`: set by `/steve:next-task` on first task execution
- `in-progress` → `checkpoint-passed`: set by `/steve:test-checkpoint` when all checks pass
- `checkpoint-passed` → `phase-completed`: set by `/steve:complete-phase` after merge
- `phase-completed` → `phase-planned`: set by `/steve:plan-phase` for the next phase (loops back)
- `checkpoint-passed` → `completed`: set by `/steve:complete-phase` when `currentPhase == totalPhases` (final phase, skips `phase-completed`)

### `docs/steve/current-phase.md`

```markdown
# Phase 3: User Authentication

## Tasks
- [x] 3.1: Set up user model with password hashing
- [x] 3.2: Implement registration endpoint
- [ ] 3.3: Implement login endpoint with JWT
- [ ] 3.4: Add auth middleware

## Test Checkpoint
- [ ] All tests pass
- [ ] API endpoints verified
- [ ] Security review passed
- [ ] Feature completeness reviewed against PRD
```

**Who marks `## Test Checkpoint` items:** The `/steve:test-checkpoint` command owns these checkboxes. When it runs, it performs each check sequentially:
1. Runs the test suite → marks "All tests pass" `[x]` or reports failures
2. Verifies API endpoints (if applicable, via curl or similar) → marks "API endpoints verified"
3. Runs the Sonnet subagent code review for security → marks "Security review passed"
4. Runs the Sonnet subagent feature completeness review → marks "Feature completeness reviewed against PRD"

All four must pass for `test-checkpoint` to set state to `checkpoint-passed`. If any fail, the command reports which items failed and leaves state as `in-progress`. The user must fix the issues and re-run `/steve:test-checkpoint`.

### Other Generated Files

| File | Created by | Purpose |
|---|---|---|
| `docs/steve/design.md` | `/steve:design` | Architecture document |
| `docs/steve/prd.md` | `/steve:prd` | Requirements + phase overview with dependencies |
| `docs/steve/completed-phases.md` | `/steve:init` (empty), populated by `/steve:complete-phase` | Archive of finished phases |
| `docs/steve/reviews/` | Review gates | Sonnet review feedback per stage |

## PRD Structure (`docs/steve/prd.md`)

The PRD is the bridge between design and execution. It defines *what* to build (requirements) and *in what order* (phases).

```markdown
# Product Requirements Document — {Project Name}

## Product Overview
Brief description of the product, its purpose, and target users.

## Success Criteria
Measurable outcomes that define when the product is complete.
- [ ] Criterion 1
- [ ] Criterion 2
- ...

## Requirements

### Functional Requirements
- FR-1: {Description}
- FR-2: {Description}
- ...

### Non-Functional Requirements
- NFR-1: {Description} (e.g., performance, security, scalability)
- ...

## Phase Overview

### Phase 1: {Name}
**Scope:** {What this phase delivers as a vertical slice}
**Dependencies:** None
**Estimated tasks:** 3-6
**Delivers:** {User-facing outcome}

### Phase 2: {Name}
**Scope:** {What this phase delivers}
**Dependencies:** Phase 1
**Estimated tasks:** 3-6
**Delivers:** {User-facing outcome}

### Phase N: {Name}
...

## Phase Dependency Graph
{Which phases depend on which, and which can potentially be parallelized in future versions}

## Out of Scope
Features explicitly excluded from this version.
```

Each phase in the overview is a **vertical slice** — it delivers a complete, testable feature from data model through API to UI (if applicable).

## Review Gates

Reviews are automatic, inline, and self-correcting. They are not a separate step — they're part of the command's execution.

### How It Works

1. Command produces its output (e.g., `design.md`)
2. Steve announces: *"Running Sonnet review for alignment..."*
3. Spawns a Sonnet subagent using Claude Code's **Agent tool** with `model: "sonnet"`
4. The subagent receives a dynamically composed prompt containing:
   - The document to review
   - The upstream document(s) to review against
   - Specific review criteria for that stage
5. Subagent returns a verdict: **APPROVED** or **ISSUES FOUND** (with specifics)
6. If issues found → Steve fixes them automatically in the current context, saves the updated output
7. Review feedback is persisted in `docs/steve/reviews/`
8. Final output is presented to the user

### Review Scope Per Command

| Command | Review Checks | Reviewed Against |
|---|---|---|
| `/steve:design` | Completeness, internal contradictions, missing security considerations, feasibility | User's stated requirements from conversation |
| `/steve:prd` | All design features covered, no scope drift, phases are vertical slices, dependencies make sense | `design.md` |
| `/steve:plan-phase` | Tasks cover full phase scope, nothing dropped, correct granularity (3-6 tasks) | Phase definition in `prd.md` |
| `/steve:test-checkpoint` | Code implements all tasks in the phase, no partial implementations, no features silently skipped | `current-phase.md` + `prd.md` |

### Review Loop Limit

Max 3 auto-fix iterations per review. If still failing after 3, Steve surfaces the issues to the user for guidance rather than looping indefinitely.

### Document Reviews vs. Code Reviews

Review gates behave differently depending on what is being reviewed:

- **Document reviews** (`design`, `prd`, `plan-phase`): The auto-fix loop applies. Steve can regenerate or edit the document to address the reviewer's findings, then re-submit for review. This is safe because documents are fully under Steve's control.
- **Code reviews** (`test-checkpoint`): The auto-fix loop does **NOT** apply. The Sonnet subagent reviews the code for feature completeness and returns its findings. If issues are found (e.g., a feature was partially implemented or silently skipped), Steve reports them to the user as a list of gaps. Steve does **not** attempt to auto-fix code — the user decides whether to run additional `/steve:next-task` iterations or address the gaps manually. State remains `in-progress` until the user re-runs `test-checkpoint` and the review passes.

### Persisted Review Output

Review feedback is saved to `docs/steve/reviews/` with consistent naming:
- `design-review.md`
- `prd-review.md`
- `phase-{N}-plan-review.md`
- `phase-{N}-checkpoint-review.md`

These files allow `/steve:resume` to reference what was caught and fixed in previous sessions.

## CLI (`steve`)

The Steve CLI is a bash script installed at `~/.local/bin/steve`. It supports two subcommands: `init` and `update`.

### Installation Layout

```
~/.local/bin/steve                    # CLI binary
~/.steve/                             # Steve home directory
├── version                           # Current version string (e.g., "0.1.0")
└── commands/                         # Source slash command files
    ├── steve:init.md
    ├── steve:design.md
    ├── steve:prd.md
    ├── steve:plan-phase.md
    ├── steve:next-task.md
    ├── steve:test-checkpoint.md
    ├── steve:complete-phase.md
    └── steve:resume.md
```

### `steve init`

Bootstraps a project by copying slash commands and launching Claude Code.

```bash
#!/usr/bin/env bash
STEVE_HOME="${HOME}/.steve"
STEVE_VERSION=$(cat "${STEVE_HOME}/version" 2>/dev/null || echo "unknown")

# Check claude CLI exists
if ! command -v claude &> /dev/null; then
    echo "Error: Claude Code CLI not found. Install it first."
    exit 1
fi

# Check git repo
if ! git rev-parse --is-inside-work-tree &> /dev/null 2>&1; then
    read -p "No git repository found. Initialize one? (y/n): " init_git
    if [[ "$init_git" == "y" ]]; then
        git init
    else
        echo "Error: Steve requires a git repository. Please initialize one and try again."
        exit 1
    fi
fi

# Copy/overwrite slash commands into project
mkdir -p .claude/commands
cp -f "${STEVE_HOME}/commands/"steve:*.md .claude/commands/
echo "Steve v${STEVE_VERSION} — commands installed to .claude/commands/"

# Launch Claude Code with the init command
claude "/steve:init"
```

**Key behavior:** `steve init` always copies the latest slash commands from `~/.steve/commands/` into the project's `.claude/commands/`, overwriting any existing versions. This ensures the project always gets the version of Steve that is installed on the user's system.

### `steve update`

Checks for a new version and updates both the CLI and slash commands.

**When run globally (outside a project):**
1. Checks the Steve GitHub repo for the latest release (via `gh api` or `curl`)
2. Compares against `~/.steve/version`
3. If newer version exists: downloads and replaces `~/.local/bin/steve`, `~/.steve/commands/`, and `~/.steve/version`
4. If already up to date: announces *"Steve v{version} is already the latest."*

**When run inside a project folder:**
1. Performs the global update (above)
2. Additionally copies the updated slash commands into the project's `.claude/commands/`, overwriting the existing ones
3. Announces: *"Project commands updated to Steve v{version}."*

This means a developer can run `steve update` inside any Steve-managed project to get the latest commands without re-running `init`.

### `/steve:init` (Slash Command)

Runs inside Claude Code as an interactive conversation (after the CLI has already copied the commands):

1. **Project Q&A:** name, description, architecture (fullstack/backend/frontend/CLI/library/monorepo), backend stack, frontend stack, database, testing methodology (TDD/BDD), git preferences (branch prefix, commit style)
2. **Dynamic generation:** creates `CLAUDE.md` with stack-appropriate patterns, conventions, security defaults, and design patterns — no templates, all generated in-context
3. **State setup:** creates `docs/steve/config.json` with all choices + `workflowState: "initialized"`, `docs/steve/completed-phases.md` (empty), `docs/steve/reviews/` directory
4. **Git:** generates stack-appropriate `.gitignore`, commits initial scaffold

## Design Phase Intelligence

The `/steve:design` command produces a comprehensive architecture document that includes:

### Design Patterns

Best patterns are prescribed based on the chosen stack:

- **Backend:** Repository → Service → Controller, middleware composition, shared error handling, reusable validators, dependency injection where appropriate
- **Frontend:** Atomic design (atoms → molecules → organisms), shared hooks, reusable form components, layout components, common UI patterns

Patterns are dynamically determined based on the stack chosen during init.

### Monolith vs. Microservices

For complex applications, the design phase explicitly assesses whether the project should be:

- **Monolith** — recommended for simpler apps, small teams, early-stage products
- **Microservices** — recommended when there are independent scaling concerns, separate deployment needs, or distinct bounded contexts (e.g., auth service, payment service, notification service)

The decision is documented with rationale and shapes everything downstream: file structure, deployment strategy, data ownership, API boundaries, and inter-service communication patterns.

### Reusable Components

The design identifies where reusable components should be built:

- Shared UI components (buttons, forms, modals, layouts)
- Common backend utilities (auth middleware, error handlers, validators)
- Shared types/interfaces across frontend and backend
- Database helpers and query builders

### Security Defaults (OWASP)

Every project includes security considerations:

1. Input validation — server-side on all endpoints
2. Authentication — bcrypt passwords, JWT tokens
3. Authorization — access control on every endpoint
4. SQL injection prevention — parameterized queries/ORM only
5. XSS prevention — output escaping, CSP headers
6. CSRF protection
7. Secure headers
8. Secrets management — environment variables only
9. Dependency security audits
10. Error handling — no stack traces in production

## `/steve:complete-phase` — Detailed Behavior

This command handles the transition between phases. It runs after `test-checkpoint` sets state to `checkpoint-passed`.

### Steps

1. **Verify state** — confirm `workflowState == "checkpoint-passed"`
2. **Archive the phase** — append the current phase to `docs/steve/completed-phases.md` in this format:

```markdown
## Phase {N}: {Name} — Completed {YYYY-MM-DD}

### Summary
{Brief description of what was delivered}

### Tasks Completed
- {task 1}
- {task 2}
- ...
```

3. **Merge the branch** — ask the user for confirmation: *"Merge `{branchPrefix}-{N}-{short-name}` to main? (yes/no)"*
   - If yes: run `git checkout main && git merge {branch-name}` (standard merge, not squash — preserves task-level commit history)
   - If no: leave the branch as-is, still advance the state
4. **Append to completedPhases** — add the current phase number to the `completedPhases` array in `config.json`
5. **Determine if final phase** — if `currentPhase == totalPhases`:
   - Set `workflowState` to `completed`
   - `currentPhase` remains at `totalPhases` (not incremented)
   - Announce: *"All phases complete. Project is done."*
6. **If not final phase:**
   - Set `currentPhase` to `currentPhase + 1`
   - Set `workflowState` to `phase-completed`
   - Clear `docs/steve/current-phase.md` (will be regenerated by next `plan-phase`)
   - Announce: *"Phase {N} complete. Run `/steve:plan-phase` to start Phase {N+1}."*
   - Suggest running `/clear` if context usage is high

### Branch Lifecycle

- **Phase number source:** `/steve:plan-phase` reads `currentPhase` from `config.json` to determine which phase to plan. On first invocation (state = `specced`), `prd` sets `currentPhase` to 1. On subsequent invocations (state = `phase-completed`), `complete-phase` has already incremented `currentPhase`. The user does not pass a phase number — it is always inferred from state.
- **Created by:** `/steve:plan-phase` — runs `git checkout -b {branchPrefix}-{N}-{short-name}` where `{N}` is `currentPhase` from `config.json` and `{short-name}` is derived from the phase title in `prd.md` (lowercased, spaces replaced with hyphens, max 30 chars)
- **If branch already exists** (e.g., resuming after a crash): `/steve:plan-phase` checks out the existing branch instead of creating a new one
- **Pushed to remote:** not automatic — Steve operates locally. User pushes when ready
- **Deleted after merge:** not automatic — user manages branch cleanup

## Smart Resume (`/steve:resume`)

The killer feature — intelligent context recovery that goes beyond reading state files.

### What It Does

1. **Read state files** — `config.json`, `current-phase.md`, `completed-phases.md`, `design.md`, `prd.md`
2. **Scan git history** — recent commits on the current branch, compare against tasks marked `[x]` in `current-phase.md`
3. **Detect manual changes** — run `git status` and `git diff` to check for:
   - Uncommitted changes (working tree is dirty)
   - Commits not associated with any task in `current-phase.md` (user worked outside Steve)
   - Files modified that aren't expected for the current phase
   - If manual changes are detected, report them and ask the user whether to incorporate or discard
4. **Assess context budget** — use heuristics to estimate remaining context capacity:
   - **Fresh session** (just started or just ran `/clear`): assume full budget → suggest `/steve:next-task`
   - **Mid-session** (resumed after some work): count the number of state files read + git history scanned during resume. If resume itself required reading many large files (design.md, prd.md, multiple review files), warn that context is partially consumed → suggest completing one task, then `/clear`
   - **Late session** (multiple tasks already completed in this session, tracked by counting task transitions in the conversation): suggest `/clear` after the current task
   - This is a best-effort heuristic, not an exact measurement. Claude Code does not expose token counts directly.
5. **Review history** — reference review feedback from `docs/steve/reviews/` to surface what was previously caught and fixed
6. **Present next action** — clear recommendation of what to do next based on current `workflowState`

### When To Use

- After `/clear` (primary use case)
- Starting a new session on an existing project
- After being away from the project for a while

## Testing Methodology

Chosen during `/steve:init` (TDD or BDD) and enforced throughout.

### TDD Flow (in `/steve:next-task`)

1. Write test (expect red)
2. Run test (confirm red)
3. Write implementation
4. Run test (confirm green)
5. Refactor if needed
6. Commit: test first, then implementation

### BDD Flow (in `/steve:next-task`)

1. Write feature spec (Gherkin or equivalent)
2. Write step definitions
3. Implement feature code
4. Run and verify pass
5. Commit per feature

## Git Workflow

Opinionated with user-configurable overrides set during init.

### Defaults

- **Branch naming:** `{branchPrefix}-{N}-{short-name}` (e.g., `phase-1-bootstrap`)
- **Commit style:** Conventional commits — `feat(phase-N):`, `test(phase-N):`, `fix(phase-N):`
- **Never commit to main directly** — always merge via branch after confirmation
- **Run tests before committing**

### Overrides

Users can customize during `/steve:init`:
- Branch prefix (default: `phase`)
- Commit style (conventional, freeform, etc.)

## Error Handling

Steve handles predictable failure modes gracefully.

### State File Errors

- **`config.json` missing or corrupt:** Command refuses to run with message: *"Steve state not found. Run `steve init` or `/steve:init` to set up this project."* Does not attempt to recover or create a partial config.
- **`current-phase.md` missing when expected:** Command refuses with message: *"No active phase found. Run `/steve:plan-phase` to start a phase."*
- **`design.md` or `prd.md` missing when referenced:** Command refuses with message pointing to the prerequisite command.

### Git Errors

- **Merge conflict during `/steve:complete-phase`:** Steve auto-resolves. Since Claude Code authored both the phase branch and main, it has full context to resolve conflicts.
  1. Steve announces: *"Merge conflict detected. Resolving automatically..."*
  2. Claude Code reads the conflicting files, understands the intent from both sides (phase tasks from `current-phase.md` + main branch state)
  3. Resolves conflicts and stages the resolved files
  4. Runs the test suite to verify the resolution
  5. If tests fail → Claude Code fixes the tests/code (it wrote both), re-runs until green (max 3 attempts)
  6. If still failing after 3 fix attempts → surfaces to user: *"Could not auto-resolve after 3 attempts. Please review the conflicts manually."* State remains `checkpoint-passed`
  7. If tests pass → completes the merge commit and continues with `complete-phase`
  - **Idempotency:** On re-run, `complete-phase` checks if the current phase is already present in the `completedPhases` array in `config.json`. If so, it skips the archive step and does not double-increment `currentPhase`. It resumes from the merge step.
  - **Already-merged detection:** Before attempting the merge, `complete-phase` checks if the phase branch is already merged into main (via `git branch --merged main`). If already merged, it skips the merge with a notice: *"Branch already merged into main. Advancing state."*
- **Branch already exists during `/steve:plan-phase`:** Checks out the existing branch instead of creating a new one. Announces: *"Branch `{name}` already exists. Checking out existing branch."*
- **Dirty working tree:** Commands that create branches (`plan-phase`) warn about uncommitted changes and ask the user to commit or stash first.

### Task Errors

- **`/steve:next-task` with no remaining unchecked tasks:** Announces: *"All tasks in this phase are complete. Run `/steve:test-checkpoint` to verify the phase."* Does not create phantom tasks.
- **`/steve:test-checkpoint` with unchecked tasks:** Lists the incomplete tasks and refuses to proceed. Hard gate, no override.

### Review Gate Errors

- **Subagent fails to return a verdict:** Treat as a single failed iteration. Retry up to 3 times. If still failing, surface to user: *"Review could not complete. Please review the output manually."*
